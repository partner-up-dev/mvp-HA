# Database Migration Workflow Plan

## Goal

Improve database migration development and deployment workflow for `apps/backend` with these constraints:

- keep using Drizzle ORM as schema source of truth
- split schema migration and data migration
- use one migration pipeline across local, staging, and production
- use one migrations table to track every applied migration
- run migrations exactly once per deploy via a dedicated one-off CI/CD job
- take a Postgres advisory lock before applying migrations
- wrap each migration in a transaction when possible
- keep local reset-based, keep staging/production forward-only
- enforce deterministic SQL artifacts in PRs

## Current Observations

1. Drizzle currently generates SQL into `apps/backend/drizzle` from `apps/backend/src/entities/*.ts`.
2. The repo exposes `db:generate` and `db:migrate`, but there is no `db:reset`, `db:seed`, or dedicated migration runner.
3. Existing migrations already mix schema and data changes in the same SQL files.
4. Deploy CI in `.github/workflows/backend-fc-deploy.yml` builds and deploys the FC app, but does not run database migrations.
5. FC packaging intentionally excludes migration artifacts and dev tooling, so running migrations inside the deployed app package is the wrong default here.
6. The backend already uses `pg_try_advisory_xact_lock(...)` in the job runner, so there is a repository precedent for lock-based single-run coordination.

## Chosen Design

### Why this design

Use GitHub Actions as the dedicated migration executor, not FC startup and not per-replica app boot.

Reasoning:

- it matches the existing deploy control plane
- it guarantees one migration attempt per deploy workflow
- it avoids shipping migration tooling into the runtime package
- it avoids startup races across FC instances
- it keeps production execution deterministic from checked-in SQL

### Migration model

Keep Drizzle for schema authoring and SQL generation, but stop relying on `drizzle-kit migrate` as the deploy-time executor.

Instead, introduce a custom migration runner that:

1. reads committed SQL artifacts from two folders:
   - `apps/backend/drizzle/` for schema migrations generated by Drizzle
   - `apps/backend/data-migrations/` for hand-authored forward-only data migrations
2. merges both folders into one ordered migration stream
3. records all applied migrations in one table, for example `public.app_migrations`
4. takes a Postgres advisory lock before scanning or applying anything
5. applies each pending migration exactly once
6. runs each migration in its own transaction unless the file explicitly opts out

This keeps schema and data migrations separated physically, but unified operationally.

### Migration identity and ordering

Use a single global sortable prefix across both folders.

Examples:

- `apps/backend/drizzle/0006_add_notification_deliveries.sql`
- `apps/backend/data-migrations/0007_backfill_notification_statuses.sql`
- `apps/backend/drizzle/0008_drop_legacy_columns.sql`

Rules:

- ordering is lexical by the numeric prefix
- prefix values are unique across both folders
- the migrations table stores a unique id derived from folder + filename
- schema and data migrations are never separated by environment

### Transaction policy

Default behavior: one transaction per migration file.

Support an explicit file header for exceptions:

- `-- migration: no-transaction`

That is needed for operations that PostgreSQL disallows inside a transaction, such as some concurrent index operations.

Runner lint rules:

- fail if a migration file contains `CONCURRENTLY` but does not include `-- migration: no-transaction`
- fail if more than one transaction-mode header is present
- treat transaction headers as case-sensitive documented contract, not an informal comment

### Abort and recovery policy

This workflow is forward-only in staging and production, so the operational story is abort-and-fix, not rollback-to-old-schema.

Rules:

1. if a migration fails, the migration job stops and the app deploy job does not run
2. transactional migrations roll back automatically at the file boundary
3. non-transactional migrations must be written to be restart-safe or split into smaller phases
4. recovery happens through a new forward migration or a targeted manual remediation, never through reset in staging or production

Practical consequences:

- additive schema changes should be preferred first
- destructive contract changes must be delayed until the new code has already been proven in production
- long-running backfills should be resumable and safe to re-run

For production incidents, the documented playbook is:

1. abort deploy on migration failure
2. inspect whether the failed file committed anything
3. if nothing committed, fix the migration and rerun
4. if partial effects exist from a non-transactional step, apply an explicit forward remediation migration
5. only deploy app code after the migration pipeline is green again

### Environment policy

#### Local development

Local development becomes reset-based only:

1. drop database
2. recreate database
3. run the full unified migration pipeline from zero
4. run local seed

Provide explicit commands for this flow and block it outside local development.

### Seed policy

Seed data is not part of the production migration ledger.

Instead, introduce a dedicated seed runner for local development and optional ephemeral environments with these rules:

1. seeds must be idempotent
2. seeds must be versioned as files in execution order
3. seeds must use stable natural keys or deterministic ids, never random inserts that duplicate on rerun
4. seeds must be safe to run after `db:reset` and safe to rerun manually
5. staging and production do not run the local seed pipeline by default

Proposed folder:

- `apps/backend/seeds/`

Proposed file naming:

- `0001_base-config.sql`
- `0002_demo-users.sql`

Seed execution model:

- the seed runner executes all seed files in lexical order
- each file must be self-idempotent using `insert ... on conflict do update`, `insert ... on conflict do nothing`, or equivalent upsert logic
- if we need tracking, use a separate `app_seed_runs` table for observability only, not as a skip ledger

Reasoning:

- migration history should represent irreversible environment evolution
- seed history should represent reproducible environment bootstrapping
- mixing the two makes local reset awkward and production rules blurry

#### Staging and production

Staging and production are forward-only:

- no reset command
- no drop/recreate path
- only apply new pending migrations
- if a migration fails, deployment stops

### Production discipline

Production changes follow expand/backfill/contract:

1. expand: additive schema change, compatible with old and new code
2. backfill: data migration after expand, still forward-only
3. contract: remove old columns or constraints only after the app no longer depends on them

This is a policy and review rule, not something the runner can infer automatically.

## Planned Repository Changes

### 1. Migration runner and commands

Add a small backend script layer, likely under `apps/backend/src/scripts/db/`:

- `migrate.ts`
- `reset.ts`
- `seed.ts`
- shared helpers for lock acquisition, file discovery, checksums, and per-file execution

Update package scripts:

- keep `db:generate` for Drizzle schema SQL generation
- replace `db:migrate` so it calls the custom unified runner
- add `db:reset`
- add `db:seed`
- add `db:next-migration <folder>`
- optionally add `db:check-artifacts` for CI

Behavioral rules:

- `db:migrate` is forward-only and used by CI/CD
- `db:reset` is local-only and runs drop/create + migrate + seed
- `db:seed` is idempotent and may be rerun locally without reset
- `db:next-migration <folder>` prints the next available global numeric prefix for either `drizzle` or `data-migrations`

Helper command behavior:

- scan both `apps/backend/drizzle/` and `apps/backend/data-migrations/`
- find the highest numeric prefix already in use across both folders
- print the next zero-padded prefix only, so it can be used in shell scripts or copy/paste workflows
- fail if `<folder>` is not one of the supported migration folders

### 2. Migration folders and policy

Keep:

- `apps/backend/drizzle/`

Add:

- `apps/backend/data-migrations/`
- `apps/backend/seeds/`

Policy:

- `drizzle/` contains only generated schema SQL
- `data-migrations/` contains hand-authored data SQL or TS-driven SQL execution if really needed
- `seeds/` contains idempotent local bootstrap data, not production migration logic
- schema changes must ship with generated SQL artifacts in the same PR
- data migrations must be explicit files, never ad-hoc CI logic

### 3. Single migrations table

Add a migration ledger table managed by the custom runner.

Proposed shape:

- `id` text primary key
- `kind` text check (`schema` or `data`)
- `checksum` text not null
- `applied_at` timestamptz not null default now()
- `duration_ms` integer not null

The runner will:

- create the table if missing
- refuse to run if an already-applied migration file changed checksum
- use this table as the only applied-history source for both schema and data migrations

### 4. GitHub Actions changes

#### PR validation workflow

Add a backend PR check workflow that:

1. installs dependencies
2. runs `pnpm --filter @partner-up-dev/backend db:generate`
3. fails if `apps/backend/drizzle/` or Drizzle meta artifacts are modified by the generation step
4. optionally validates migration ordering rules across `drizzle/` and `data-migrations/`

This enforces deterministic SQL artifacts in PRs instead of generating them at deploy time.

#### Deploy workflow

Refactor `.github/workflows/backend-fc-deploy.yml` so deploy is gated by a dedicated `migrate` job.

Target sequence:

1. `migrate`
2. `deploy`
3. `publish version / alias` for production branch

`migrate` job responsibilities:

- install dependencies
- run the custom migration runner against the target `DATABASE_URL`
- use workflow/job concurrency plus DB advisory lock
- fail fast on checksum drift or SQL failure

This satisfies “exactly once per deploy” operationally and still protects against accidental duplicate runners at the DB level.

Update:

Document:

## Implementation Steps

1. Introduce the custom unified migration runner with advisory lock, checksum validation, transaction policy, and single ledger table.
2. Add runner lint rules for transaction headers, including `CONCURRENTLY` detection.
3. Introduce the seed runner with ordered idempotent seed files and local-only reset flow.
4. Split operational commands into `db:generate`, `db:migrate`, `db:reset`, `db:seed`, and `db:next-migration`.
5. Create `apps/backend/data-migrations/` and `apps/backend/seeds/` with filename/order rules.
6. Add PR CI to regenerate Drizzle artifacts and fail on dirty diff.
7. Refactor deploy workflow so migration runs before FC deploy.
8. Update deployment and backend docs, including the abort-and-fix playbook and migration authoring rules.
9. Run build verification.

## Trade-offs

### Accepted

- We keep Drizzle for schema generation, but not as the direct deploy-time migrator.
- We add a small amount of custom migration infrastructure to get one ledger table and a split schema/data model.
- Developers must manage one global migration order across two folders.
- We reduce numbering collisions by providing a helper command, but concurrent branches can still conflict at merge time and must be rebased.

### Rejected

- separate migration folders per environment: guaranteed drift
- migration generation inside CI/CD: nondeterministic and hard to review
- app-startup migrations: race-prone in serverless and violates one-off requirement
- reset in staging or production: destroys environment trust
- independent schema ledger and data ledger: conflicts with the single-table requirement

## Risks To Watch

1. Existing mixed schema/data Drizzle SQL files will remain in history. The new split applies to new migrations going forward.
2. Contract migrations merged too early can still break live code; this must be controlled in review using expand/backfill/contract discipline.
3. Non-transactional SQL needs explicit opt-out markers, otherwise the runner should default to transactional safety.
4. Idempotent seeds require discipline around stable keys; tables without natural unique keys may need explicit deterministic identifiers or supporting unique constraints.
5. `db:next-migration` reduces local collisions but cannot eliminate cross-branch numbering conflicts before merge; CI should still validate uniqueness.

## Approval Gate

If approved, implementation will follow this plan with one notable opinionated choice:

- migrations will run in GitHub Actions as a dedicated one-off job before FC deploy, not inside FC itself

That is the cleanest fit for this repository’s current packaging and deployment model.
