# PLAN-STEP-B — 活动区结构重做计划（非对称节奏版）

## 1. 目标对齐（来自 PROPOSAL Step B）

本计划只处理 Step B：活动区结构重建。

目标不是“把 4 个活动显示出来”，而是：

- 明确主次层级（Lead Unit + Sub Units）。
- 打破重复卡片模式，形成非对称视觉节奏。
- 提升移动端扫读效率，减少“每块看起来都一样”的疲劳感。

## 2. 当前实现与目标差距

当前 `HomeEventSectionV2` 虽然替换了旧双屏结构，但仍存在这些问题：

- 主位卡 + 3 张次位卡仍然是同类卡片语法的重复。
- `SubCardA` 和 `SubCardB` 差异不足，阅读节奏仍然线性。
- 4 个活动没有形成“信息密度梯度”，都在争同一级注意力。
- 布局上缺少“一个重锤 + 两个轻击 + 一个收口”的节奏结构。

结论：当前版本只能算“去 Slide”，还不是 Step B 要求的“结构重建”。

## 3. 重做后的结构蓝图

## 3.1 版面骨架（纵向连续 section）

活动区改成 4 个连续结构块：

1. `Live Strip`：热度与即时信号（短条，不占主视觉）
2. `Lead Block`：主位活动（大图/大标题/强动作）
3. `Mosaic Block`：两块不等分拼接（7:5 或 5:7），承载 2 个活动
4. `Focus Block`：最后 1 个活动做“收口块”，并连接 `All Events Entry`

这 4 块之间间距不等，避免机械网格感。

## 3.2 非对称规则（硬约束）

- 禁止出现“连续两块同骨架”。
- 禁止四个活动都使用同级标题字号。
- 禁止四个活动都使用同类 CTA 表达。
- 至少有 1 块使用横向信息条（meta rail），至少有 1 块使用纵向信息堆栈。

## 4. 组件重构方案

不再以 `LeadCard + SubCardA + SubCardB` 作为最终结构，改成“布局块组件”：

- `HomeEventLiveStrip.vue`
- `HomeEventLeadBlock.vue`
- `HomeEventMosaicBlock.vue`
- `HomeEventFocusBlock.vue`
- `HomeEventAllEntry.vue`

说明：

- 活动单元数据仍由 `useHomeEventUnitData` 提供。
- 组件按“布局角色”组织，不按“活动类型”组织。
- 同一活动可根据优先级进入不同块位（不是固定映射到固定卡片模板）。

## 5. 数据绑定策略（Step B 范围内）

- 继续使用 `leadUnit + subUnits`。
- `subUnits` 在结构层重排为：
  - `mosaicPrimary`
  - `mosaicSecondary`
  - `focusUnit`
- 重排规则先采用“稳定排序 + 日内不抖动”。

## 6. 样式与交互要求（Step B 内）

- 仅实现结构相关样式差异，不在 Step B 内扩写新动效系统。
- 保留现有按压反馈与入场能力，但不新增更多动画逻辑。
- 需要局部横向滚动时，使用原生 `scroll-snap`，不引入 JS 滑动库。

## 7. 任务拆分

### Task B1：拆除当前重复卡片框架

- 从 `HomeEventSectionV2.vue` 移除 `SubCardA/SubCardB` 网格渲染。
- 清理“统一卡片容器”布局逻辑。

### Task B2：实现 4 段结构块

- 新增 5 个布局块组件并接入活动单元数据。
- 完成主次信息密度差异（标题、文案、meta、CTA）。

### Task B3：重排策略接入

- 在 section 层完成 `subUnits -> mosaic/focus` 映射。
- 确保无数据时仍有稳定兜底块位。

### Task B4：移动端视觉回归

- 覆盖 360 / 390 / 430 宽度。
- 检查连续滚动中的节奏变化是否明显。

## 8. 验收标准（Step B 完成定义）

- 活动区不再使用“一个主卡 + 三个同类次卡”的重复模式。
- 存在 4 个结构块且相邻块视觉语法不同。
- 任意两块活动内容的主标题层级不完全一致。
- 用户首屏下滑 1 次后能明显感知到版式切换。
- 构建通过：`pnpm --filter @partner-up-dev/frontend build`。

## 9. 风险与控制

- 风险：结构变化过大导致实现周期失控。
  - 控制：先做块级骨架与数据绑定，再做视觉精修。
- 风险：过度个性化导致可维护性下降。
  - 控制：统一块级 props 契约，避免组件内硬编码业务规则。
- 风险：排序变化引起用户定位困难。
  - 控制：保持“主位稳定，次位日内稳定”策略。

## 10. 交付顺序建议

1. 先提交骨架与组件迁移（无视觉细节）。
2. 再提交视觉差异化与布局调优。
3. 最后提交 Step B 验收修补（边距、文案长度、断点问题）。
