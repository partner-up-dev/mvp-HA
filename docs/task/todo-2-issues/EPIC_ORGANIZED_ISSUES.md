## 🛡️ Epic 1: Trust & Safety (核心产品合规与安全边界)

### 1. [Backend] 提供法务条款版本管理与拦截 API

- **类型**：Feature
- **优先级**：P0
- **标签**：`backend`, `legal`, `auth`, `api`
- **目标**：提供版本化的协议查询、状态写入，以及对未同意用户的 API 中间件拦截。
- **验收标准**：
  - [ ] 数据库表结构需支持存储用户签署的特定版本号及时间戳。
  - [ ] 提供 GET 接口获取当前最新版本的 ToS/Privacy URL 及版本号。
  - [ ] 提供 POST 接口记录用户的同意状态。
  - [ ] 在关键的敏感/创建接口层添加 Middleware，若未同意则返回特定错误码 (例如 451 Unavailable For Legal Reasons)。

### 2. [Frontend] 用户隐私与服务条款页面及拦截确认交互

- **类型**：Feature
- **优先级**：P0
- **标签**：`frontend`, `legal`, `ux`
- **目标**：纯前端视角的合规落地：提供静态协议渲染、弹窗提示以及异常捕获跳转。
- **验收标准**：
  - [ ] 提供专门的路由 (`/tos`, `/privacy`) 用于全屏/嵌入展示条款。
  - [ ] 在全局 Request 拦截器中处理 451 错误，弹出强制同意弹窗或跳转至签署页。
  - [ ] 用户初次登录/核心转化操作前，展示清晰的“勾选同意”交互。

### 3. [FullStack] 约束搭子规则：强制 minPartners > 1 的前后台闭环

- **类型**：Bug/Rule
- **优先级**：P0
- **标签**：`backend`, `frontend`, `domain-rule`
- **目标**：修补核心业务漏洞，禁止创建单人请求。在结构化创建、编辑、NL（自然语言）解析、后台管理四条路径统一约束。
- **验收标准**：
  - [ ] Backend: Schema 校验层拒绝 `minPartners <= 1` 或空值的提交。
  - [ ] Frontend: 表单验证加上相应的红字提示。
  - [ ] LLM Parse: 调整 Prompt 保证大模型提取或兜底时给出的 `minPartners` 至少为 2。
  - [ ] Admin: 兼容历史错误数据，读取不崩溃，但编辑保存时必须满足最新规则。

## 🚀 Epic 2: Fulfillment (服务履约与闭环)

### 4. [Frontend] Anchor PR 非独立微信环境登录流程死胡同兜底

- **类型**：Feature
- **优先级**：P0
- **标签**：`frontend`, `auth`, `wechat-fallback`
- **目标**：修复在非微信环境 (例如普通手机浏览器、小红书内置浏览器) 中，Anchor 页面无法走微信授权导致的死胡同流失。
- **验收标准**：
  - [ ] 探测当前环境是否支持微信登录。
  - [ ] 若不支持，将主要的“加入”按钮替换为引导态（例如弹窗展示小程序码、公众号二维码，或提示复制链接到微信打开）。
  - [ ] 无任何前端白屏或按钮无响应的“假死”状态。

### 5. [Admin] 运营控制台：搭子请求执行视图与结果反馈面板

- **类型**：Feature
- **优先级**：P0
- **标签**：`admin`, `frontend`
- **目标**：在后台管理系统中，为运营人员提供处理场馆预订及搭子请求的专用视图。
- **验收标准**：
  - [ ] 列表视图：聚合展示待处理的请求、代理手机号、目标场地。
  - [ ] 操作面板：提供“标记成功”、“标记失败”的按钮，失败需必填原因。

### 6. [Backend] 服务类：实现基于审核状态机的用户结果触达

- **类型**：Feature
- **优先级**：P0
- **标签**：`backend`, `notification`, `wechat`
- **目标**：承接营运控制台的操作，将结果闭环通知给终端用户。
- **验收标准**：
  - [ ] 状态机：支持 `Pending` -> `Success` / `Failed` 流转。
  - [ ] 触达：调用微信模板消息机制（或短信兜底），给对应用户下发成功凭证或失败退款/建议通知。

## 📈 Epic 3: Growth & Conversion UX (转化增长与渠道分发)

### 7. [Frontend] 首页落地引导文案与首屏架构调优

- **类型**：Enhancement
- **优先级**：P1
- **标签**：`frontend`, `marketing`, `ux`
- **目标**：强化核心价值主张，减少跳出率。突出“无登录/无隐私收集”等用户顾虑痛点。
- **验收标准**：
  - [ ] Hero 区域文案更新，进行重点信息突出。
  - [ ] 首部操作区及快速查看区 UI/UX 改善。
  - [ ] 加入关键跳跃点的点击埋点。

### 8. [Frontend] 分享模块 PRShareSection 的视觉密度重构

- **类型**：Enhancement
- **优先级**：P1
- **标签**：`frontend`, `share`, `ui`
- **目标**：提升分享弹窗/模块的信息密度与操作效率。
- **验收标准**：
  - [ ] 将 WeChat、小红书、系统原生分享入口进行统一风格的布局。
  - [ ] 统一并美化获取 Poster、短链接时的加载中(Loading)和失败反馈态。
  - [ ] 确保在标准尺寸(iOS/Android 主流机型)屏幕下不出现滚动条或截断。

### 9. [Spike/Frontend] 尝试唤起小红书 App 端内深链草稿

- **类型**：Research/Feature
- **优先级**：P1
- **标签**：`frontend`, `xiaohongshu`, `mobile`
- **目标**：调研并尽最大可能实现在浏览器点击一键跳转至小红书 App 发布页。
- **验收标准**：
  - [ ] 提供技术调研方案（Schema URL 限制、系统拦截说明）。
  - [ ] 如果可行，实装点击按钮唤起小红书并在粘贴板置入文案的流程；如果不可行，优化兜底的“保存图片+复制文案”的切换动画与提示文案体验。

## ⚙️ Epic 4: Engineering (工程体验与监控基建)

### 10. [Perf] 明确并优化关键路径的 Mobile LCP

- **类型**：Performance
- **优先级**：P2
- **标签**：`frontend`, `perf`
- **目标**：解决移动端加载瓶颈，提升用户打开体感。
- **验收标准**：
  - [ ] 定义当前的性能基点（在真实低端机型上的首屏 LCP，如 >3s）。
  - [ ] 实施关键渲染路径优化、资源拆包懒加载，移除无用依赖。
  - [ ] 调优完成后，目标移动端(3G/4G网络下)首屏内容可见(LCP)稳定在设定阈值范围内。

### 11. [Chore] CI/CD 第二阶段提效与制品缓存

- **类型**：Chore
- **优先级**：P2
- **标签**：`devops`, `ci`
- **目标**：缩短流水线耗时并提升稳定性。
- **验收标准**：
  - [ ] 建立或优化 pnpm store 的自动化缓存。
  - [ ] 并行化不互相依赖的 Job（如前端 Build 和后端 Build）。
  - [ ] 整体 Pipeline 平均执行时间比当前缩短至少 20%。
